## 🔍 第二阶段：根因分析

### 2.1 分析方法选择

使用 **5 Whys + 鱼骨图** 混合方法进行根因分析。

---

### 2.2 5 Whys 分析

**第 1 个 Why**：为什么 Web 界面显示"未授权"？
**回答**：因为网关令牌验证失败

**第 2 个 Why**：为什么网关令牌验证失败？
**回答**：因为 Web 界面没有提供有效的令牌

**第 3 个 Why**：为什么 Web 界面没有提供有效令牌？
**回答**：因为有两种可能：
  A. 用户没有在 Control UI 中粘贴令牌
  B. 没有使用 tokenized dashboard URL

**第 4 个 Why**：为什么用户不知道需要这些步骤？
**回答**：因为文档和界面提示不够清晰

**第 5 个 Why**：为什么会出现这个情况？
**回答**：因为 OpenClaw 的安全设计要求显式授权，但用户引导不足

**根本原因**：
1. **技术根因**：缺少显式的授权令牌传递
2. **流程根因**：用户缺少清晰的授权步骤引导
3. **文档根因**：Web 界面授权方法说明不足

## 💡 第三阶段：开发多个潜在解决方案

### 3.1 解决方案头脑风暴

| 编号 | 方案名称 | 简要描述 |
|-----|---------|---------|
| **方案 A** | Control UI 粘贴令牌 | 在 Web 界面设置中粘贴已有 operator 令牌 |
| **方案 B** | 使用 Tokenized URL | 查找并使用带令牌的完整 URL |
| **方案 C** | 修改配置文件 | 把正确令牌写入 openclaw.json |
| **方案 D** | 重新配对设备 | 生成新的令牌和配对 |
| **方案 E** | 临时禁用认证 | 降低安全性以快速恢复（不推荐） |

---

### 3.2 方案详细描述

#### 方案 A：Control UI 粘贴令牌（推荐）

**描述**：指导用户在 Web 界面设置中粘贴已有令牌

**步骤**：
1. 找到 operator 令牌：`46b9943ad8704211a9a18409d1c36f11`
2. 在 Web 界面找设置/配置/Control 菜单
3. 找到 Gateway Token 输入框
4. 粘贴令牌并保存
5. 刷新页面验证

**所需资源**：
- 用户操作（无需代码更改）
- 已有令牌（已找到）

---

#### 方案 B：使用 Tokenized URL

**描述**：查找并访问带令牌参数的完整 URL

**步骤**：
1. 查看 OpenClaw 启动日志
2. 寻找类似 `http://127.0.0.1:18789/?token=xxxxx` 的 URL
3. 直接在浏览器访问该完整 URL
4. 应该自动授权连接

**所需资源**：
- 用户操作（无需代码更改）
- 需要检查启动日志

---

#### 方案 C：修改配置文件

**描述**：将正确的令牌写入 openclaw.json 配置文件

**步骤**：
1. 备份原配置文件
2. 将 `temp_token_123` 替换为 `46b9943ad8704211a9a18409d1c36f11`
3. 保存配置文件
4. 重启 OpenClaw 网关
5. 刷新 Web 界面

**所需资源**：
- 文件系统访问权限
- 需要重启服务
- 配置文件修改

---

#### 方案 D：重新配对设备

**描述**：通过重新配对流程生成新的授权令牌

**步骤**：
1. 查看 OpenClaw 配对相关命令
2. 运行配对流程
3. 生成新的令牌和配对
4. 使用新令牌连接

**所需资源**：
- OpenClaw CLI 访问
- 配对流程知识
- 可能需要重新授权

---

#### 方案 E：临时禁用认证（不推荐）

**描述**：修改配置以跳过认证要求（仅用于测试）

**步骤**：
1. 修改 gateway auth 模式
2. 设为无需认证
3. 重启服务
4. 验证连接

**风险**：
- 🔴 安全风险
- 🔴 不推荐生产环境
